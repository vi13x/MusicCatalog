# Теория по 4 лабе

## 1. Проверяемые и непроверяемые исключения

### Проверяемые исключения
Проверяемые исключения (`checked exceptions`) наследуются от `Exception`, но не от `RuntimeException`.
Компилятор заставляет либо обработать их через `try/catch`, либо пробросить через `throws`.

Примеры:
- `IOException`
- `SQLException`
- `ParseException`

Когда применяются:
- когда ошибка ожидаема;
- когда вызывающий код реально может на нее отреагировать;
- когда восстановление после ошибки возможно.

### Непроверяемые исключения
Непроверяемые исключения (`unchecked exceptions`) наследуются от `RuntimeException`.
Компилятор не требует явно их обрабатывать.

Примеры:
- `NullPointerException`
- `IllegalArgumentException`
- `IllegalStateException`
- пользовательские `NotFoundException`, `BadRequestException`

Когда применяются:
- при ошибках бизнес-логики;
- при некорректных входных данных;
- при программных ошибках;
- в REST API, когда исключение потом централизованно переводится в HTTP-ответ.

### Вывод
В Spring REST-проектах чаще используют непроверяемые исключения, а затем обрабатывают их централизованно через `@ControllerAdvice`.

## 2. @ControllerAdvice, @ExceptionHandler, @Valid

### @ControllerAdvice
`@ControllerAdvice` нужен для глобальной обработки исключений.
Он позволяет не писать одинаковый `try/catch` в каждом контроллере.

Если используется `@RestControllerAdvice`, ответы автоматически возвращаются в JSON.

Что дает:
- единый формат ошибок;
- чистые контроллеры;
- централизованную обработку исключений;
- одинаковое поведение всех endpoint.

### @ExceptionHandler
`@ExceptionHandler` связывает тип исключения с методом-обработчиком.

Пример логики:
- `NotFoundException` -> `404 Not Found`
- `MethodArgumentNotValidException` -> `400 Bad Request`
- `DataIntegrityViolationException` -> `409 Conflict`
- `Exception` -> `500 Internal Server Error`

### @Valid
`@Valid` запускает bean validation для DTO.

Обычно используется так:
- в контроллере на `@RequestBody`;
- ограничения задаются в DTO;
- если данные невалидны, Spring выбрасывает `MethodArgumentNotValidException`.

Примеры ограничений:
- `@NotBlank`
- `@NotNull`
- `@Size`
- `@Positive`
- `@Min`
- `@Max`

### Итог
Связка `@Valid` + DTO-аннотации + `@ControllerAdvice` обеспечивает:
- проверку входных данных;
- единый JSON ошибки;
- корректные HTTP-статусы.

## 3. Логирование

### Зачем нужно логирование
Логирование позволяет:
- видеть, что делает приложение;
- анализировать ошибки;
- отслеживать время выполнения операций;
- понимать историю событий;
- упрощать отладку и сопровождение.

### Уровни логирования
- `TRACE` — максимально подробная техническая информация.
- `DEBUG` — отладочная информация.
- `INFO` — обычные значимые события.
- `WARN` — подозрительные ситуации, но приложение еще работает.
- `ERROR` — серьезные ошибки выполнения.

### Основные библиотеки

#### SLF4J
Это фасад логирования.
Он дает единый API для логирования, а реальная реализация подключается отдельно.

#### Logback
Это реализация логирования, которая по умолчанию используется в Spring Boot.
Через нее настраиваются appenders, шаблоны, уровни и ротация логов.

#### Log4j / мосты
Иногда сторонние библиотеки используют Log4j.
Тогда применяется мост в SLF4J, чтобы все логи шли через единый механизм.

### Ротация логов
Если складывать все в один файл, лог будет бесконечно расти.
Поэтому используют ротацию.

Ротация логов позволяет:
- ограничивать размер файла;
- делить логи по дате и размеру;
- архивировать старые логи;
- ограничивать общий объем хранения.

Пример настроек ротации:
- `maxFileSize`
- `maxHistory`
- `totalSizeCap`
- шаблон имени архивного файла

### Вывод
Логирование в проекте должно быть настроено так, чтобы:
- в консоль и файл попадали нужные события;
- были разные уровни логирования;
- старые логи архивировались автоматически.

## 4. AOP, Aspect, JoinPoint, Advice

### Что такое AOP
AOP (`Aspect-Oriented Programming`) — аспектно-ориентированное программирование.
Оно нужно для вынесения сквозной функциональности из бизнес-кода.

Сквозная функциональность:
- логирование;
- аудит;
- безопасность;
- метрики;
- замер времени выполнения;
- транзакции.

### Aspect
`Aspect` — это класс, в котором описано сквозное поведение.

Пример:
- логировать время выполнения всех сервисных методов.

### JoinPoint
`JoinPoint` — точка в выполнении программы, куда можно "подключиться".

В Spring AOP чаще всего join point — это вызов метода бина.

### Pointcut
`Pointcut` — правило, которое определяет, какие join points нужно перехватить.

Пример:
- все public методы в пакете `service`.

### Advice
`Advice` — действие, выполняемое относительно вызова метода.

Основные виды:
- `@Before` — до вызова;
- `@After` — после вызова;
- `@AfterReturning` — после успешного завершения;
- `@AfterThrowing` — если метод завершился исключением;
- `@Around` — полный контроль над вызовом.

### Почему для времени выполнения используют @Around
`@Around` позволяет:
- выполнить код до вызова метода;
- вызвать исходный метод;
- замерить время;
- залогировать успех или ошибку;
- снова выбросить исключение при необходимости.

## 5. Логирование через AOP vs логирование в коде

### Логирование через AOP
Плюсы:
- нет копипаста;
- единый стиль логирования;
- удобно для технических сквозных задач;
- легко применить ко многим сервисам сразу.

Минусы:
- логика менее очевидна при чтении кода;
- нужно понимать прокси и аспекты;
- сложнее отладка для новичка.

Когда использовать:
- для замера времени;
- для технической трассировки;
- для аудита одинаковых операций.

### Логирование в коде
Плюсы:
- максимально явно;
- легко видеть бизнес-смысл события;
- удобно для предметных сценариев.

Минусы:
- дублирование;
- загрязнение сервисов и контроллеров;
- сложнее поддерживать единый подход.

Когда использовать:
- когда сообщение зависит от бизнес-контекста;
- когда важно явно показать доменное событие.

### Вывод
Лучший подход:
- техническое логирование — через AOP;
- бизнес-события — в коде явно.

## 6. Паттерны проектирования

### Что такое паттерн
Паттерн проектирования — это типовое решение повторяющейся задачи проектирования.
Это не готовый код, а схема решения.

### Типы паттернов

#### 1. Порождающие
Отвечают за создание объектов.

Примеры:
- `Singleton`
- `Builder`
- `Factory Method`

#### 2. Структурные
Описывают, как классы и объекты компонуются друг с другом.

Примеры:
- `Adapter`
- `Proxy`
- `Facade`
- `Decorator`

#### 3. Поведенческие
Описывают взаимодействие объектов и распределение обязанностей.

Примеры:
- `Strategy`
- `Observer`
- `Command`
- `Template Method`

### Минимум по одному паттерну каждого типа

#### Порождающий — Builder
Используется, когда объект имеет много полей и его удобно собирать пошагово.

#### Структурный — Proxy
В Spring AOP сервисные бины оборачиваются в proxy, через который добавляется дополнительное поведение.

#### Поведенческий — Strategy
Подходит, когда есть несколько алгоритмов решения одной задачи, а конкретный выбирается во время выполнения.

#### Singleton
Один экземпляр объекта на все приложение.
В Spring бины по умолчанию singleton.
Например, сервисы и конфигурационные классы создаются как один экземпляр на весь контекст.

### Что можно говорить по проекту
По проекту можно упомянуть:
- `Proxy` — Spring AOP;
- `Singleton` — Spring beans;
- `Builder` — если используется Lombok `@Builder`;
- `Facade` — сервисный слой как упрощенный вход к репозиториям;
- `Adapter` — mapper между entity и DTO;
- `Strategy` — выбор режима выполнения алгоритма.

## 7. Рефакторинг

### Что такое рефакторинг
Рефакторинг — это изменение внутренней структуры кода без изменения внешнего поведения.

### Зачем он нужен
- уменьшить дублирование;
- улучшить читаемость;
- упростить поддержку;
- разделить ответственности;
- подготовить код к расширению;
- уменьшить вероятность ошибок в будущем.

### Примеры рефакторинга
- вынести обработку ошибок в `GlobalExceptionHandler`;
- вынести валидацию в DTO;
- вынести время выполнения сервисов в аспект;
- удалить лишний или мертвый код;
- выделить mapper-классы;
- разбить большой метод на несколько маленьких.

### Правила хорошего рефакторинга
- менять код маленькими шагами;
- не смешивать большой рефакторинг и новую бизнес-логику;
- после каждого шага проверять сборку;
- подтверждать изменения тестами или ручной проверкой.

## 8. Короткий вывод по лабе
Эта лаба показывает зрелый подход к backend-разработке:
- ошибки обрабатываются централизованно;
- входные данные валидируются;
- все endpoint могут возвращать единый формат ошибки;
- логирование настроено через уровни и ротацию;
- сквозное логирование времени вынесено в AOP;
- структура проекта опирается на паттерны проектирования;
- код можно улучшать через рефакторинг без изменения поведения.

Материал по темам: исключения, @ControllerAdvice, @ExceptionHandler, @Valid, логирование, AOP, паттерны проектирования и рефакторинг.

1. Проверяемые и непроверяемые исключения

Проверяемые исключения

Проверяемые исключения (checked exceptions) наследуются от Exception, но не от RuntimeException. Компилятор заставляет либо обработать их через try/catch, либо пробросить через throws.

Примеры: IOException, SQLException, ParseException.

Они применяются, когда ошибка ожидаема и вызывающий код может на нее отреагировать.

Непроверяемые исключения

Непроверяемые исключения (unchecked exceptions) наследуются от RuntimeException. Компилятор не требует обрабатывать их явно.

Примеры: NullPointerException, IllegalArgumentException, IllegalStateException, пользовательские NotFoundException и BadRequestException.

В REST API обычно используют именно непроверяемые исключения и потом централизованно переводят их в HTTP-ответы.

2. @ControllerAdvice, @ExceptionHandler, @Valid

@ControllerAdvice

@ControllerAdvice нужен для глобальной обработки исключений. Он позволяет не писать одинаковый try/catch в каждом контроллере.

@RestControllerAdvice делает то же самое, но возвращает ответы сразу в JSON.

  • единый формат ошибок;
  • чистые контроллеры;
  • централизованная обработка;
  • одинаковое поведение всех endpoint.

@ExceptionHandler

@ExceptionHandler связывает тип исключения с методом обработки.

Исключение Типичный статус
NotFoundException 404 Not Found
MethodArgumentNotValidException 400 Bad Request
DataIntegrityViolationException 409 Conflict
Exception 500 Internal Server Error

@Valid

@Valid запускает bean validation для DTO. Ограничения задаются аннотациями в самих DTO.

Основные аннотации: @NotBlank, @NotNull, @Size, @Positive, @Min, @Max.

Если данные невалидны, Spring выбрасывает MethodArgumentNotValidException, а глобальный обработчик переводит её в 400.

3. Логирование

Зачем нужно логирование

  • для отладки;
  • для диагностики ошибок;
  • для анализа поведения приложения;
  • для контроля времени выполнения;
  • для сопровождения проекта.

Уровни логирования

Уровень Назначение
TRACE максимально подробная техническая информация
DEBUG отладочная информация
INFO обычные значимые события
WARN подозрительные ситуации
ERROR серьезные ошибки

Основные библиотеки

  • SLF4J — фасад логирования;
  • Logback — реализация логирования по умолчанию в Spring Boot;
  • Log4j bridges — мосты для сторонних библиотек.

Ротация логов

Ротация логов нужна, чтобы файлы не росли бесконечно. Обычно настраиваются:

  • maxFileSize — максимальный размер файла;
  • maxHistory — сколько архивов хранить;
  • totalSizeCap — ограничение общего объема;
  • архивирование старых логов.

4. AOP, Aspect, JoinPoint, Advice

Что такое AOP

AOP (Aspect-Oriented Programming) — аспектно-ориентированное программирование. Оно позволяет вынести сквозную функциональность из бизнес-кода.

Сквозная функциональность: логирование, аудит, безопасность, метрики, замер времени, транзакции.

Основные термины

Термин Смысл
Aspect класс с аспектной логикой
JoinPoint точка выполнения программы, куда можно подключиться
Pointcut правило выбора join point
Advice действие до, после или вокруг вызова метода

Виды Advice

  • @Before
  • @After
  • @AfterReturning
  • @AfterThrowing
  • @Around

Для логирования времени выполнения лучше всего подходит @Around.

5. Логирование через AOP vs логирование в коде

Подход Плюсы Минусы
Через AOP нет копипаста, единый стиль, удобно для технических задач логика менее очевидна, нужно понимать proxy и аспекты
В коде явно видно контекст, удобно для бизнес-событий дублирование, загрязнение кода

Лучший подход: техническое логирование — через AOP, бизнес-логи — в коде явно.

6. Паттерны проектирования

Что такое паттерн

Паттерн проектирования — это типовое решение повторяющейся архитектурной задачи. Это не готовый код, а схема решения.

Типы паттернов

  • Порождающие — создание объектов: Singleton, Builder, Factory Method.
  • Структурные — композиция объектов: Adapter, Proxy, Facade, Decorator.
  • Поведенческие — взаимодействие объектов: Strategy, Observer, Command, Template Method.

Минимум по одному каждого типа + Singleton

  • порождающий — Builder;
  • структурный — Proxy;
  • поведенческий — Strategy;
  • отдельно — Singleton.

Что можно говорить по проекту

  • Proxy — Spring AOP;
  • Singleton — Spring beans по умолчанию;
  • Builder — если используется Lombok @Builder;
  • Adapter — mapper между entity и DTO;
  • Facade — сервисный слой;
  • Strategy — выбор алгоритма.

7. Рефакторинг

Что такое рефакторинг

Рефакторинг — это изменение внутренней структуры кода без изменения внешнего поведения.

Зачем он нужен

  • уменьшить дублирование;
  • улучшить читаемость;
  • упростить поддержку;
  • разделить ответственности;
  • подготовить код к расширению.

Примеры рефакторинга

  • вынести обработку ошибок в GlobalExceptionHandler;
  • вынести валидацию в DTO;
  • вынести замер времени в аспект;
  • удалить мертвый код;
  • выделить mapper-классы.

Правила хорошего рефакторинга

  • делать изменения маленькими шагами;
  • после изменений проверять сборку;
  • не смешивать большой рефакторинг и новую бизнес-логику;
  • подтверждать изменения тестами или ручной проверкой.

8. Итог

Эта лаба показывает зрелый backend-подход:

  • ошибки обрабатываются централизованно;
  • входные данные валидируются;
  • есть единый формат ошибок;
  • логирование настроено по уровням и с ротацией;
  • сквозное техническое логирование вынесено в AOP;
  • проект использует паттерны проектирования;
  • код можно улучшать через рефакторинг без изменения поведения.

1. ПРОВЕРЯЕМЫЕ И НЕПРОВЕРЯЕМЫЕ ИСКЛЮЧЕНИЯ

Проверяемые исключения (checked exceptions) наследуются от Exception, но не от RuntimeException.
Компилятор заставляет либо обработать их через try/catch, либо пробросить через throws.

Примеры:
- IOException
- SQLException
- ParseException

Используются, когда ошибка ожидаема и вызывающий код может на нее отреагировать.

Непроверяемые исключения (unchecked exceptions) наследуются от RuntimeException.
Компилятор не требует их явно обрабатывать.

Примеры:
- NullPointerException
- IllegalArgumentException
- IllegalStateException
- пользовательские NotFoundException и BadRequestException

В REST API обычно удобнее использовать непроверяемые исключения и централизованно переводить их в HTTP-ответы.

2. @CONTROLLERADVICE, @EXCEPTIONHANDLER, @VALID

@ControllerAdvice нужен для глобальной обработки исключений.
Он позволяет не писать try/catch в каждом контроллере.

@RestControllerAdvice делает то же самое, но ответы автоматически возвращаются в JSON.

Преимущества:
- единый формат ошибок;
- чистые контроллеры;
- централизованная обработка;
- одинаковое поведение всех endpoint.

@ExceptionHandler связывает тип исключения с методом обработки.

Примеры:
- NotFoundException -> 404
- MethodArgumentNotValidException -> 400
- DataIntegrityViolationException -> 409
- Exception -> 500

@Valid запускает bean validation для DTO.
Ограничения ставятся в самом DTO:
- @NotBlank
- @NotNull
- @Size
- @Positive
- @Min
- @Max

Если входные данные невалидны, Spring выбрасывает MethodArgumentNotValidException.

Итог:
@Valid + DTO-аннотации + @ControllerAdvice дают корректную проверку входа и единый JSON ошибок.

3. ЛОГИРОВАНИЕ

Логирование нужно для:
- отладки;
- диагностики ошибок;
- отслеживания поведения приложения;
- анализа времени выполнения;
- сопровождения системы.

Уровни логирования:
- TRACE — максимально детально
- DEBUG — отладочная информация
- INFO — обычные события
- WARN — подозрительные ситуации
- ERROR — серьезные ошибки

Основные библиотеки:
- SLF4J — фасад логирования
- Logback — реализация по умолчанию в Spring Boot
- мосты Log4j -> SLF4J — для сторонних библиотек

Ротация логов нужна, чтобы лог-файлы не росли бесконечно.
Обычно настраиваются:
- maxFileSize
- maxHistory
- totalSizeCap
- архивирование старых логов

4. AOP, ASPECT, JOINPOINT, ADVICE

AOP — аспектно-ориентированное программирование.
Оно позволяет вынести сквозную функциональность из бизнес-кода.

Сквозная функциональность:
- логирование
- аудит
- безопасность
- метрики
- замер времени
- транзакции

Aspect — класс с аспектной логикой.
JoinPoint — точка выполнения программы, куда можно подключиться.
Pointcut — правило выбора таких точек.
Advice — действие до, после или вокруг вызова метода.

Виды Advice:
- @Before
- @After
- @AfterReturning
- @AfterThrowing
- @Around

Для логирования времени выполнения лучше всего подходит @Around.

5. ЛОГИРОВАНИЕ ЧЕРЕЗ AOP VS ЛОГИРОВАНИЕ В КОДЕ

Логирование через AOP:
Плюсы:
- нет копипаста
- единый стиль
- удобно для технических задач
- легко применить ко многим методам

Минусы:
- логика менее очевидна
- нужно понимать proxy и аспекты

Использовать для:
- замера времени
- технической трассировки
- одинакового аудита

Логирование в коде:
Плюсы:
- явно видно контекст
- удобно для бизнес-событий

Минусы:
- дублирование
- загрязнение кода

Использовать для:
- доменных событий
- бизнес-значимых сообщений

Вывод:
технические логи лучше выносить в AOP, а бизнес-логи оставлять в коде.

6. ПАТТЕРНЫ ПРОЕКТИРОВАНИЯ

Паттерн проектирования — типовое решение повторяющейся архитектурной задачи.

Типы паттернов:

1. Порождающие
Нужны для создания объектов.
Примеры:
- Singleton
- Builder
- Factory Method

2. Структурные
Нужны для композиции классов и объектов.
Примеры:
- Adapter
- Proxy
- Facade
- Decorator

3. Поведенческие
Нужны для описания взаимодействия объектов.
Примеры:
- Strategy
- Observer
- Command
- Template Method

Минимум по одному:
- порождающий: Builder
- структурный: Proxy
- поведенческий: Strategy
- отдельно: Singleton

Singleton:
один экземпляр на все приложение.
В Spring бины по умолчанию singleton.

Что можно говорить по проекту:
- Proxy — Spring AOP
- Singleton — Spring beans
- Builder — если используется Lombok @Builder
- Adapter — mapper между entity и DTO
- Facade — сервисный слой
- Strategy — выбор алгоритма

7. РЕФАКТОРИНГ

Рефакторинг — изменение внутренней структуры кода без изменения внешнего поведения.

Зачем нужен:
- убрать дублирование
- улучшить читаемость
- упростить поддержку
- разделить ответственности
- подготовить код к расширению

Примеры:
- вынести обработку ошибок в GlobalExceptionHandler
- вынести валидацию в DTO
- вынести замер времени в аспект
- удалить мертвый код
- выделить mapper-классы

Правила:
- делать маленькими шагами
- после изменений проверять сборку
- не смешивать большой рефакторинг и новую бизнес-логику

8. ИТОГ

Эта лаба показывает зрелый backend-подход:
- ошибки обрабатываются централизованно
- входные данные валидируются
- есть единый формат ошибок
- логирование настроено по уровням и с ротацией
- сквозное техническое логирование вынесено в AOP
- проект использует паттерны проектирования
- код можно улучшать через рефакторинг без изменения поведения
